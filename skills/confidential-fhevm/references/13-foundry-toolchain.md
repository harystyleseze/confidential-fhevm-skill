## 1. Why Foundry is the canonical track now

Zama's official `fhevm-react-template` uses Foundry + `forge-fhevm` as the contract toolchain. The Solidity API surface (`@fhevm/solidity 0.11.1`, `FHE.*`, `ZamaEthereumConfig`, `FHE.fromExternal`, etc.) is identical between Foundry and Hardhat; only the surrounding tooling differs:

- Tests inherit `FhevmTest` instead of using a `fhevm` mock object
- Deploys are `.s.sol` scripts invoked via `forge script`, not `hardhat-deploy` TypeScript files
- Dependencies are managed via `soldeer` (`forge soldeer install`), not npm
- The local FHEVM stack runs as a real cleartext host on anvil, not a JS-side mock

If you're working from the official template today, you're on Foundry. The Hardhat track (`references/02-project-setup.md`, the `@fhevm/hardhat-plugin`-based testing in `references/07-testing-guide.md`) is still valid for existing projects but is no longer the default.

## 2. Project structure (the official template)

```
my-dapp/
├── package.json                          ← workspace root (pnpm)
├── pnpm-workspace.yaml
├── scripts/
│   ├── chain.sh                          ← pnpm chain  → anvil + cleartext host
│   ├── deploy-localhost.sh               ← pnpm deploy:localhost
│   ├── deploy-sepolia.sh                 ← pnpm deploy:sepolia
│   └── generateTsAbis.ts                 ← pnpm generate → emits frontend ABI files
└── packages/
    ├── foundry/
    │   ├── foundry.toml                  ← solc 0.8.27, evm cancun, runs 800, bytecodeHash none
    │   ├── remappings.txt                ← managed by soldeer
    │   ├── soldeer.lock                  ← lockfile for deterministic builds
    │   ├── src/                          ← your contracts
    │   ├── test/                         ← *.t.sol  inheriting FhevmTest
    │   ├── script/                       ← Deploy*.s.sol  invoked by deploy-*.sh
    │   ├── dependencies/                 ← installed by `forge soldeer install`
    │   ├── out/                          ← build artifacts (compiled ABIs)
    │   └── broadcast/                    ← run-latest.json per deploy; consumed by generateTsAbis.ts
    └── nextjs/
        ├── app/                          ← Next.js 15 App Router
        ├── hooks/                        ← your SDK v3 hooks
        ├── contracts/                    ← auto-generated by pnpm generate
        │   ├── <Name>.ts                 ← non-local chains (tracked)
        │   └── <Name>.local.ts           ← chainId 31337 overlay (gitignored)
        └── utils/contract.ts             ← deploymentFor(Contract, chainId)
```

`pnpm generate` (and the postinstall hook) walks `packages/foundry/broadcast/` and `packages/foundry/out/` to emit one `<Name>.ts` + one `<Name>.local.ts` per deployed contract. Consumers import `<Name>` and the local overlay merges transparently.

## 3. The `forge-fhevm` test harness

```solidity
import {FhevmTest} from "forge-fhevm/FhevmTest.sol";

contract MyTest is FhevmTest {
    function setUp() public override {
        super.setUp();                            // deploys ACL, Executor, Decryption, InputVerifier
        // your contract setup goes here
    }
}
```

`setUp()` stands up the entire FHEVM host stack as real Solidity contracts on the in-memory anvil. Tests call the same `FHE.*` API your contract uses, against the same cleartext executor. No JS mock layer, no relayer, no KMS.

The test base class exposes helpers (one ciphertext at a time):

```solidity
// encryption — each call produces ONE handle + ONE proof
(externalEbool   encB, bytes memory pB) = encryptBool(true, alice, contract);
(externalEuint8  e8 , bytes memory p8) = encryptUint8(42, alice, contract);
(externalEuint64 e64, bytes memory p64) = encryptUint64(1_000n, alice, contract);
// ... encryptUint16/32/128/256, encryptAddress

// decryption (cleartext-mode, test-only)
uint64 v = decrypt(myEuint64);                    // direct cleartext read

// user-decrypt EIP-712 path (mocks the SDK's useUserDecrypt flow)
bytes memory sig = signUserDecrypt(USER_PK, contractAddress);
uint256 clear = userDecrypt(handle, user, contractAddress, sig);

// public-decrypt proof construction (for testing FHE.checkSignatures, see §5)
bytes memory proof = buildDecryptionProof(handles, abiEncodedCleartexts);
```

## 4. Per-ciphertext proofs (a deliberate design choice)

In real frontend code, the SDK's `useEncrypt` can batch multiple ciphertexts into one input proof:

```typescript
const enc = await encrypt.mutateAsync({
  values: [{ value: true, type: "ebool" }, { value: 100n, type: "euint64" }],
  contractAddress, userAddress,
});
// enc.handles[0], enc.handles[1] both validated by enc.inputProof
```

`forge-fhevm`'s cleartext mode validates one ciphertext per proof — there is no batched-input helper. If your contract takes a single shared `bytes inputProof` for multiple ciphertexts, it works fine in production but **cannot be tested** end-to-end in cleartext mode.

The skill's recommended pattern: design contract functions to take one proof per ciphertext.

```solidity
function vote(
    uint256 proposalId,
    externalEbool   encryptedIsYes, bytes calldata isYesProof,
    externalEuint64 encryptedWeight, bytes calldata weightProof
) external onlyMember {
    ebool   isYes  = FHE.fromExternal(encryptedIsYes,  isYesProof);
    euint64 weight = FHE.fromExternal(encryptedWeight, weightProof);
    ...
}
```

From the SDK side, this means two `mutateAsync` calls instead of one batched call (each call signs one proof). The trade-off is one extra round-trip per submission, in exchange for full Foundry testability. For most apps this is the right call. See `references/04-encrypted-io.md` for the alternative single-proof pattern (still usable; just harder to test in cleartext mode).

## 5. Public-decryption testing with `buildDecryptionProof`

In the Hardhat mock harness, `FHE.checkSignatures` is effectively Sepolia-only because the proof comes from KMS. forge-fhevm's `buildDecryptionProof(bytes32[] handles, bytes abiEncodedCleartexts)` builds a proof signed by the cleartext executor's mock KMS that `FHE.checkSignatures` accepts:

```solidity
function test_finalizeWithValidProof() public {
    // ... after votes are cast and deadline passes ...
    voting.requestReveal(0);

    (euint64 yesEnc, euint64 noEnc) = voting.encryptedTallies(0);
    bytes32[] memory handles = new bytes32[](2);
    handles[0] = euint64.unwrap(yesEnc);
    handles[1] = euint64.unwrap(noEnc);

    bytes memory cleartexts = abi.encode(uint64(100), uint64(25));
    bytes memory proof = buildDecryptionProof(handles, cleartexts);

    voting.finalize(0, 100, 25, proof);                    // works in cleartext mode
}
```

This is the single biggest reason to use Foundry: you can test the complete public-decryption ceremony in your unit test suite. Hardhat's mock plugin does not expose an equivalent helper today.

## 6. Soldeer dependencies

`forge-fhevm` and its peers are installed via [soldeer](https://soldeer.xyz), Foundry's official package manager:

```bash
cd packages/foundry
forge soldeer install              # or `pnpm contracts:install` from the workspace root
```

This reads `[dependencies]` from `foundry.toml` and writes `dependencies/` + `remappings.txt`. The lockfile is `soldeer.lock`. The canonical pinned versions for FHEVM dApps today:

| Dep | Version | Notes |
| --- | --- | --- |
| `@fhevm-solidity` | `0.11.1` | the Solidity API library |
| `@encrypted-types` | `0.0.4` | external-type wrappers (`externalEuintNN`, etc.) |
| `forge-fhevm` | `git@eba2324` | test harness + cleartext host |
| `@openzeppelin-confidential-contracts` | `git@6edd293` | ERC-7984 + wrapper |
| `@openzeppelin-contracts` | `5.1.0` | base OZ |
| `@openzeppelin-contracts-upgradeable` | `5.1.0` | upgradeable OZ |
| `forge-std` | `1.14.0` | forge utilities |

`templates/foundry/foundry.toml` includes the exact `[dependencies]` block.

## 7. Cleartext mode vs. real Sepolia

| Aspect | Local (anvil + cleartext executor) | Sepolia |
| --- | --- | --- |
| FHE math | real `@fhevm/host-contracts` on chain, plaintext stored in `plaintexts(bytes32)` mapping | real homomorphic on the coprocessor |
| Relayer | none; SDK uses `RelayerCleartext` to read plaintext directly | real `RelayerWeb` over the network |
| KMS | mocked by `CleartextFHEVMExecutor` + the cleartext host's signer | real threshold KMS |
| `userDecrypt` | EIP-712 signature is validated, returns plaintext immediately | EIP-712 → relayer → KMS round-trip (1–3 s) |
| `publicDecrypt` | `buildDecryptionProof` available | proof comes from KMS over the relayer |
| `decrypt(handle)` test helper | available (reads from cleartext mapping) | does not exist |
| FHE.decrypt(...) in contracts | **never call this from production code** (AP-018) | reverts |
| Gas | unlimited (anvil) | block gas limit 16,777,216 — cap `useWriteContract` gas at ~15M |
| Wallet | anvil dev keys (0xf39F…0x9965) | your funded Sepolia wallet |

Tests that pass in cleartext mode but rely on production-only behaviour (e.g., latency assumptions, real network costs) should be marked Sepolia-only and gated.
